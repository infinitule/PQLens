## Principle: open core, separable commercial surface

The open core already does the hard, honest part: measure, discover, price,
hybridize, diagnose entropy, and produce signed compliance evidence — locally,
offline, with no account required. The commercial layer must add **operational
scale and continuity**, never gate the core capabilities behind a paywall, and
never compromise a guardrail. If a commercial feature would require weakening a
guardrail, it does not ship.

What must always stay in the free, open core:
- All CLI/library capabilities shipped in v0.1.0.
- Local, offline operation with no telemetry by default.
- The cited data tables (`algorithms.json`, `compliance.json`) and the ability to
  run/verify signed reports yourself.

## Candidate commercial modules (design sketches)

### C-1 — Managed scanning service
Hosted, authenticated scanning of larger/remote codebases and artifact registries
(container images, repos) with scheduled re-scans and diffing over time.
- **Value over core:** scale, scheduling, multi-repo aggregation, history.
- **Boundary:** the *scanner* stays open; the *hosting/orchestration/auth/storage*
  is the paid surface. A customer can always self-host the open scanner instead.
- **Guardrail watch:** uploading code is sensitive — explicit consent, data
  retention controls, and an on-prem/self-hosted option are mandatory, not optional.

### C-2 — Fleet dashboard
Aggregate inventories + cost + compliance across many services into one
quantum-readiness view: per-team risk burndown, "below-required-level" tracking,
trend lines toward CNSA 2.0 / FIPS / EU milestones.
- **Value over core:** cross-service rollup, RBAC, shareable exec reporting.
- **Boundary:** the per-service reports are produced by the open core and simply
  ingested; the dashboard/rollup/identity layer is paid.

### C-3 — Continuous-compliance SaaS
Re-run compliance evidence on every CI run / on a schedule, sign with the
customer's attested organizational key (a real KMS/HSM-backed key — replacing the
ephemeral-key caveat from D-006), alert on regressions, retain an auditable,
verifiable history of signed reports.
- **Value over core:** continuity, attested signing, alerting, retention/audit.
- **Boundary:** the report + signature format stay open and independently
  verifiable; the scheduling, KMS integration, alerting, and audit store are paid.
- **Guardrail watch:** "attested" must mean a genuine PKI/KMS key, surfaced
  honestly (no implying more assurance than the key chain provides).

## Honest boundary (open vs. paid)

| Capability | Open core (Apache-2.0) | Commercial |
|---|---|---|
| Scan / measure / hybrid / entropy / compliance | ✅ full, local | — |
| Signed report format + self-verification | ✅ | — |
| Hosted scheduling, multi-repo aggregation | — | ✅ |
| Fleet dashboard, RBAC, exec reporting | — | ✅ |
| KMS/HSM-attested signing, alerting, audit retention | — | ✅ |

## Sequencing (design only — not a commitment to build)

A defensible order that compounds value and de-risks each step:

1. **C-3 first, in its thinnest form: attested signing + history.** The open core
   already emits verifiable signed reports; the smallest paid wedge is replacing
   the ephemeral key (D-006) with a customer KMS/HSM-attested key plus a retained,
   queryable history of signed reports. Narrow, high trust value, little surface.
2. **C-1 managed scanning.** Add hosted/scheduled scanning once there is a reason
   to store results over time (history from step 1). Self-host stays first-class.
3. **C-2 fleet dashboard.** Aggregate only after multiple services produce
   reports worth rolling up; it is a read/visualization layer over 1–2.

Each step must ship with the open core still fully usable without it.

## Monetization & licensing model

- **Open core: Apache-2.0, unchanged.** No open-core feature ever moves behind a
  paywall; the paid layer only *adds* hosting/scale/continuity/identity.
- **Commercial code lives in a separate repo/package** (e.g. `pqlens-cloud`),
  depending on `pqlens` as a normal library — never a fork, never a re-license.
  This keeps the boundary auditable: anything in the open repo is free forever.
- **Avoid the open-core trap:** do not cripple the core to force upgrades. The
  test is simple — a self-hosting user must be able to reproduce every *analytic*
  result locally; they pay only to not operate it themselves.

## Pricing philosophy (placeholder, to validate with users)

- Price the **operational outcome** (continuous assurance, fleet rollups,
  attested audit trail), not the algorithms — those are open and free.
- Likely shape: per-seat or per-repo for dashboards; usage/retention-based for
  hosted scanning + signed-history storage. Generous free tier that mirrors the
  open core so the upgrade is about scale, not capability.
- Validate willingness-to-pay before building: the buyer is a security/compliance
  owner who needs *evidence over time*, which the one-shot open core does not give.

## Key risks & mitigations

| Risk | Mitigation |
|---|---|
| Open-core resentment ("they paywalled X") | Hard rule: analytics stay reproducible locally; publish the open/paid boundary table and never move a line across it. |
| Handling customer source code (C-1) | Consent-first, on-prem/self-host option, short retention, scan-in-place where possible; treat uploads as sensitive by default. |
| Over-claiming assurance (signing/compliance) | Carry D-006 honesty verbatim: "attested" means a real KMS/HSM chain; compliance stays advisory; no certification language. |
| Guardrail erosion under sales pressure | A commercial feature that needs a guardrail weakened does not ship — written into the gate below. |
| Commoditization by upstream/cloud vendors | Differentiate on the honest measurement/evidence layer + continuity, not on raw crypto (which we deliberately don't compete on). |

## Validation milestones (before any build)

- 5+ design-partner interviews confirming the "evidence over time" pain is real
  and budgeted.
- A written open/paid boundary the maintainer signs off on (the table above).
- A data-handling/consent design for hosted scanning reviewed and approved.
- Confirmation that no proposed feature trades away a guardrail.

## Explicit non-goals for the commercial layer
- Never relicense or cripple the open core to drive upgrades.
- Never add default telemetry/phone-home to the core.
- Never claim certification/assurance the implementation cannot back (the
  compliance-is-advisory and signing-honesty rules from D-006 carry over verbatim).

## Gate before any commercial build
Stop here. Before writing a line of commercial code: confirm with the maintainer
(a) the open-core boundary above, (b) data-handling/consent for hosted scanning,
and (c) that no guardrail is traded away for a feature.
